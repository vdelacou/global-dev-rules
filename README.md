# Global Dev Rules

A small, stack-agnostic reference for how a software project should work — decided before what it is built with. Three documents, read in order:

1. **[Core Values: what my writing says I stand for](core-values-one-pager.md)** — the five values the rules serve, and how they map to the eighteen pillars.
2. **[The Global Rules Every New Project Should Have](global-rules-every-new-project.md)** — the eighteen pillars, in prose.
3. **[The Global Rules: Do and Don't, with Examples](global-rules-dos-and-donts.md)** — a Do/Don't with contrasting TypeScript and Java code for every sub-concept.

The principles name no language, database, or cloud: the stack is a detail, and these rules describe how a team works rather than what it works with. The Do/Don't examples favor TypeScript and Java and the standard library, with a framework appearing only where the concept needs one. The rules are grounded in the essays under [`medium-series/`](medium-series/).

## How the docs connect

Each pillar in the prose doc links to its Do/Don't counterpart, and vice versa, so principle and code are one click apart. Start with the values, read the pillars, then jump to the code for any rule you want to see enforced.
